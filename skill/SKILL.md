# 《都市余烬》开发向导 (Dushi Dev Guide)

本 Skill 是《都市余烬》项目的标准开发指南。在执行该项目的任何开发任务前，必须严格遵循本指南中的流程与协作规则。

## 核心协作规则

《都市余烬》采用**动态数据驱动 (Data-Driven)** 架构。作为工程师（AI），你的职责是构建通用技术框架，而所有业务逻辑、文本、数值均由**制作人 (Yaahua)** 提供。

**黄金法则**：
1. **永远不要代替制作人做决定**：不要擅自为任何属性、时间、价格、惩罚等系统填写默认数值。
2. **方案先行，代码在后**：收到需求或发现问题后，必须先撰写方案文档（含问题描述、代码定位、可选方案），提交给制作人审阅。制作人确认后方可编写代码。不得跳过方案阶段直接修改代码。
3. **每一步必须审查**：在编写代码或推进流程时，必须在每个审查节点停下来，向制作人展示结果（截图、演示或文档），并等待其明确的修改意见或通过指令。
4. **按需加载上下文**：不要一次性阅读所有参考文件，根据当前所处的开发阶段，只加载对应的参考文件。

## 开发流程指南

整个项目的开发被划分为四个渐进阶段。**当前我们处于：第一阶段（基础设施搭建与 MVP 静态验证）**。

### 如何推进开发？

1. **查阅当前阶段任务**：读取 `references/dev-roadmap.md`，找到当前阶段的第一个未完成步骤。
2. **确认前置条件**：检查该步骤是否需要制作人提供数据（例如 MVP 的初始状态值）。如果需要，必须先询问制作人并等待回复。
3. **执行技术开发**：根据制作人提供的数据和架构要求，编写代码。
4. **提交审查节点**：完成该步骤的代码后，向制作人展示结果（如 UI 截图、逻辑流程说明），并询问："请问这个实现是否符合您的预期？是否有需要修改的地方？"
5. **等待指令**：在制作人回复"可以"或给出修改意见前，**绝对不要**擅自进入下一个开发步骤。

### 示例交互

**错误的做法**：
> "我为您写好了 MVP 的移动逻辑，假设每次移动消耗 10 点体能，耗时 5 分钟。接下来我开始写交易逻辑。" (违反了黄金法则 1 和 2)

**正确的做法**：
> "我已经搭建好了 UI 界面原型，请看截图。请问视觉效果是否符合方案 B 的要求？另外，为了编写移动逻辑，请您提供火车站区内平均移动的时间消耗和体能消耗数值。"

## 立即执行的下一步

如果你刚刚被唤醒或被要求"继续开发"，请首先阅读 `references/dev-roadmap.md`，确认当前进度，并向制作人汇报下一个需要执行的步骤及所需的审查。
