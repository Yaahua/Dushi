# 开发注意事项与教训总结

## 核心原则

### 1. 不要过度设计

**问题**: 用户报告「上楼回公寓」按钮失效，我设计了一个通用的 `previousLocation` 返回系统。

**错误**:
- 忽略了用户只是想修复一个具体 Bug
- 通用返回机制与场景导航设计冲突（如果哪里都能返回，就没必要规划路线了）

**正确做法**:
- 先定位具体 Bug 的根源
- 修复具体问题，不要引入额外复杂度
- 如果有设计需求，先询问制作人确认

---

### 2. 深入理解现有代码逻辑

**问题**: 买水后「上楼回公寓」按钮失效。

**原因**: `BUY_ITEM` reducer 调用了 `expireNarrativeLinks()`，导致场景链接被过期。

**教训**:
- `expireNarrativeLinks` 的设计意图是：当玩家离开场景时，过期该场景的操作链接
- 但买水是**同场景内**的操作，不应该过期链接
- 需要区分「场景内操作」和「场景切换操作」

**检查清单**:
- [ ] 修改 reducer 前，理解现有逻辑的设计意图
- [ ] 区分哪些操作应该过期链接，哪些不应该
- [ ] 场景切换（GOTO_LOCATION）→ 过期旧链接
- [ ] 同场景内操作（买东西、吃东西）→ 不过期链接

---

### 3. 准确理解需求

**问题**: 用户要求「饥饿体力的加减不要体现在对话中」。

**我的理解**: 数值变化要完全静默，不显示任何文本。

**实际需求**: 去掉数值部分，保留简洁的操作反馈。

| 需求 | 错误实现 | 正确实现 |
|------|----------|----------|
| 买水 | 完全静默 | 「你购买了一瓶矿泉水。」 |
| 吃饭 | 完全静默 | 「你吃了一顿饭。」 |
| 喝水 | 完全静默 | 「你喝了一瓶矿泉水。」 |

**教训**:
- 「去掉数值」≠ 「去掉所有反馈」
- 保留简洁的文本反馈，只去掉具体的数字变化
- 数值变化通过顶部状态栏实时反映

### 5. 代码审查清单

修改 reducer 前检查：
- [ ] 这个操作是否改变场景？→ 是则过期链接，否则不过期
- [ ] 这个操作是否需要显示反馈？→ 需要则添加简洁文本（无数值）
- [ ] 这个操作是否改变玩家状态？→ 是则更新 state，顶部状态栏会自动反映
- [ ] 操作完成后是否需要刷新场景链接？→ 是则重新添加场景描述和 links

### 6. 通用"操作后选项"机制

**设计**: 使用 `onActionComplete` 配置 + `getActionResultCard` 工具函数

**配置示例**:
```typescript
'location_id': {
  // ... 基础配置
  onActionComplete: {
    buy: { text: '购买后的描述', links: [...] },
    eat: { text: '吃饭后的描述', links: [...] },
  }
}
```

**reducer 中使用**:
```typescript
const actionCard = getActionResultCard(loc, 'buy');
narrative: actionCard
  ? [...state.narrative, { type: 'system', text: '...' }, actionCard]
  : [...state.narrative, { type: 'system', text: '...' }],
```

**优势**: 新场景只需配置，无需修改 reducer

### 7. 常见 Bug 模式

#### Bug A: 链接失效
**症状**: 点击操作按钮无响应
**可能原因**:
- `expireNarrativeLinks` 被错误调用
- 场景的 `actions` 配置缺失
- `executeCommand` 没有处理对应命令

#### Bug B: 状态不更新
**症状**: 买了东西/吃了饭，顶部状态栏数值没变
**可能原因**:
- 只更新了 narrative，没更新 state
- 命令解析直接 return，没有 dispatch action

#### Bug C: 操作后无响应
**症状**: 执行操作后没有任何反馈
**可能原因**:
- reducer 返回了原 state（没有变化）
- narrative 没有添加新条目
- `onActionComplete` 未配置且没有默认行为
