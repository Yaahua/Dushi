# 后端启动方案（FastAPI + PostgreSQL + Redis）

本文档详细规划了《都市余烬》项目后端基础设施的启动步骤。本方案旨在将当前纯前端运行的 Tick 引擎和状态管理，平滑迁移至后端驱动，实现真正的多人在线与离线消耗基础。

## 一、技术栈与目录结构

### 1. 技术栈选型

*   **Web 框架**：FastAPI（原生支持异步与 WebSocket，适合高频 Tick 推送）
*   **数据库**：PostgreSQL（存储玩家持久化状态、账单、物品）
*   **ORM**：SQLAlchemy（配合 Pydantic 进行数据校验）
*   **缓存与消息队列**：Redis（存储在线玩家会话、高频状态快照、Pub/Sub 广播）
*   **容器化**：Docker Compose（一键拉起本地开发环境的数据库与缓存）

### 2. 后端目录结构规划

建议在项目根目录下创建或重构 `server/` 目录，结构如下：

```text
server/
├── app/
│   ├── main.py              # FastAPI 应用入口
│   ├── core/                # 核心配置
│   │   ├── config.py        # 环境变量与配置
│   │   └── security.py      # 认证与安全
│   ├── api/                 # API 路由
│   │   ├── routes.py        # HTTP 接口
│   │   └── websocket.py     # WebSocket 连接管理
│   ├── engine/              # 游戏核心引擎
│   │   ├── tick.py          # City Tick 定时任务
│   │   └── calculator.py    # 属性消耗与结算逻辑
│   ├── models/              # SQLAlchemy 数据库模型
│   ├── schemas/             # Pydantic 数据校验模型
│   └── services/            # 业务逻辑层（如玩家状态管理）
├── requirements.txt         # Python 依赖清单
└── docker-compose.yml       # 本地开发环境编排文件
```

## 二、实施步骤与审查节点

后端启动将分为三个阶段进行，每个阶段完成后均需制作人审查。

### 阶段一：基础设施搭建与连接测试

**目标**：建立可运行的后端骨架，打通数据库与缓存连接。

1.  **编写 `docker-compose.yml`**：配置 PostgreSQL 和 Redis 容器，设置初始密码与端口映射。
2.  **初始化 FastAPI 项目**：编写 `main.py`，配置跨域（CORS）以允许前端访问。
3.  **建立数据库连接**：使用 SQLAlchemy 配置 PostgreSQL 连接池。
4.  **建立缓存连接**：使用 `redis-py` 配置 Redis 连接池。

**审查节点**：向制作人展示 `docker-compose up` 成功运行的截图，以及 FastAPI 自动生成的 Swagger API 文档页面（`/docs`）。

### 阶段二：WebSocket 通信与前端对接

**目标**：建立前后端双向实时通信通道。

1.  **后端 WebSocket 管理器**：在 `api/websocket.py` 中实现连接池，管理所有在线玩家的 WebSocket 会话。
2.  **前端对接**：修改前端 `GameContext.tsx`，在初始化时建立 WebSocket 连接，监听后端推送的消息。
3.  **指令透传**：前端的 `executeCommand` 不再直接修改本地状态，而是将指令（如 `goto apartment_kitchen`）通过 WebSocket 发送给后端。

**审查节点**：向制作人演示前端输入指令后，后端能成功接收并打印日志，前端能收到后端的确认回执。

### 阶段三：Tick 引擎后端化迁移

**目标**：将时间流逝与属性消耗逻辑完全交由后端接管。

1.  **实现后台 Tick 任务**：在 `engine/tick.py` 中使用 `asyncio` 创建定时任务，模拟 GT 时间的流逝。
2.  **状态结算逻辑**：将前端 `gameReducer` 中的 `GT_TICK` 逻辑（饥饿增加、生命值扣除等）迁移至后端 `calculator.py`。
3.  **状态广播**：每次 Tick 结算后，后端通过 WebSocket 将最新的 `PlayerStats` 和 `gtTime` 广播给对应的前端客户端。
4.  **前端纯渲染化**：移除前端 `GameContext` 中的 `setInterval`，前端仅负责接收后端推送的最新状态并渲染 UI。

**审查节点**：向制作人演示关闭前端浏览器后，后端时间继续流逝；重新打开浏览器后，状态与时间自动同步至最新进度。
