# 机制改进与远期规划

本文档记录制作人在第一阶段中后期审查中提出的机制层面的改进方向和远期规划要点。这些内容不属于当前阶段的紧急修复，但需要在架构设计中提前预留接口。

---

## 一、游戏入口流程重构（远期）

### 当前状态

`mvp-src/client/src/pages/Home.tsx` 中的 `GamePhase` 流程为：

```
intro → characterCreation → transit → gameplay
```

每次刷新页面都从 `intro`（边境站入境）开始，因为 `GameContext.tsx` 的 `initialState.phase` 硬编码为 `'intro'`。

### 制作人要求

角色注册与入境流程仅在账号首次创建时触发一次。后续登录应从主界面或上次下线位置继续游戏，不能又从边疆开始。

### 远期方案

接入后端后，`GamePhase` 流程将重构为：

```
login → [新玩家: intro → characterCreation → transit → gameplay]
       → [老玩家: gameplay（从数据库恢复坐标和状态）]
```

具体实现：
1. 增加 `'login'` 和 `'main_menu'` 两个新的 `GamePhase`
2. 登录成功后，后端 API 返回玩家数据（包含 `phase` 字段）
3. 如果是新玩家（数据库中无记录），进入 `intro` 流程
4. 如果是老玩家，直接进入 `gameplay`，所有状态从后端恢复
5. `initialState` 不再硬编码，由后端 API 动态提供

**实施时机**：后端基础设施（数据库 + 用户认证）就绪后。

## 二、游戏时间系统的长期解决方案

### 当前限制

游戏时间由前端 `setInterval`（每 1000ms）驱动，浏览器关闭后时间停止。这是前端模拟阶段的已知限制。

### 制作人确认

制作人已明确表示：游戏时间是一个大问题，但因为没有后端不能始终在后台计算。这个以后能解决就是了。

### 长期方案要点

1. **世界时钟**：由后端 City Tick Engine 驱动，7x24 小时运行，所有玩家共享同一世界时间
2. **个人时钟**：每个玩家的个人时间流速可能因状态不同而有差异（如睡眠加速）
3. **离线补算**：玩家上线时，后端根据离线时长一次性计算离线期间的属性变化
4. **简报系统**：离线期间发生的重要事件（如账单到期、被驱逐等）通过简报通知玩家

**前置依赖**：后端 Tick 引擎 + PostgreSQL 状态持久化 + Redis 实时同步。

## 三、属性系统扩展预留

### 当前 `PlayerStats` 接口

```typescript
interface PlayerStats {
  name: string;
  gender: '男' | '女';
  stamina: number;      // 体能
  maxStamina: number;
  hunger: number;       // 饥饿
  maxHunger: number;
  money: number;        // 金钱
  skills: { ... };      // 四维技能点
}
```

### 需要扩展的属性（待制作人定义数值后实装）

| 属性 | 字段名 | 说明 | 状态 |
| :--- | :--- | :--- | :--- |
| 生命值 | `health` / `maxHealth` | 核心生存指标，归零触发死亡/昏迷 | 待制作人提供数值 |
| 心理状态 | 待定 | 长期孤独/压力可能影响行为效率 | 远期预留 |
| 信誉值 | 待定 | 影响 NPC 态度、租房、求职 | 远期预留 |
| 犯罪记录 | 待定 | 影响区域准入、NPC 互动 | 远期预留 |

工程端将在 `PlayerStats` 接口设计上保持可扩展性，确保新属性可以无缝接入。

```

├── apartment_room（你的房间）
│   ├── apartment_kitchen（厨房）
│   └── apartment_room（含沙发交互）
```

## 五、协作流程补充

制作人在本次审查中明确重申了协作原则：

> "你现在只是一个工程师……你都不能直接进行修改，你要注意你是工程师我是制作人我最后拍板，你要给我方案。"

此原则已在 `skill/SKILL.md` 的黄金法则中有所体现，但本次制作人进一步强调了**方案先行**的工作模式：

1. 工程师发现问题或收到需求后，**先撰写方案文档**
2. 方案文档提交到仓库供制作人审阅
3. 制作人确认或修改方案后，工程师方可执行代码修改
4. 代码修改完成后，再次提交审查

此流程适用于所有非紧急的功能开发和 Bug 修复。
