# Bug 修复与 UI 改进方案

本文档基于 `mvp-src/` 目录下的实际代码，逐项记录制作人在第一阶段中后期审查中提出的 Bug 和 UI 改进需求。每个问题均标注了涉及的源文件和具体代码位置，并提供工程师的技术方案供制作人审批。

**所有方案均需制作人确认后方可执行。**

---

## Bug-01：叙事流文字应始终居中于屏幕视口

**问题描述**：当前叙事流中的文字不断追加到底部，玩家视线需要持续跟踪页面底端。制作人要求：无论怎么滚动，最新一段文字始终出现在屏幕正中间，上一段文字向上推移。

**涉及文件**：
- `mvp-src/client/src/components/NarrativeStream.tsx`（第 24-26 行：`scrollIntoView` 滚动到底部）
- `mvp-src/client/src/index.css`（第 163-177 行：`scene-enter` 淡入动画，无居中锚定机制）

**当前行为**：`NarrativeStream.tsx` 使用 `bottomRef.current?.scrollIntoView({ behavior: 'smooth' })` 将视口滚动到最底部的 `<div ref={bottomRef} />`。所有叙事条目从上到下线性排列，新内容出现在列表末尾。

**修复方案**：

**制作人已确认采用方案 A 的变体**：
1. **视口居中锚定**：将 `NarrativeStream` 的外层容器配置为使"当前焦点段落"始终在屏幕正中显示，历史段落向上推移。新段落出现时，当前段落平滑上移，新段落从底部淡入到中心位置。
2. **历史记录回滚查看**：
   - **当前阶段需求**：所有历史记录完全保留，玩家可以向上滚动（回滚）查看所有历史文字。
   - **未来阶段需求（远期变更）**：只允许回看游戏内"一天"的历史记录，更早的内容将从主叙事流中移除，玩家需通过单独的"日志"系统查看。

*注：当前阶段先实现完全回滚查看，并在架构上为未来的"一天"截断和日志系统预留接口。*

## Bug-02：历史交互选项点击后应失效

**问题描述**：叙事流中出现的超链接（如 `[去厨房]`、`[下楼]`、`[买矿泉水 (¥2)]`）在被点击执行后依然可以反复点击，导致逻辑错乱（例如已经离开公寓后，仍然可以点击历史记录中的"去厨房"）。

**涉及文件**：
- `mvp-src/client/src/components/NarrativeStream.tsx`（第 37-49 行：`line.links` 渲染为 `<button>` 并直接调用 `executeCommand`）
- `mvp-src/client/src/contexts/GameContext.tsx`（第 139 行：`ADD_NARRATIVE` action 仅追加，无失效标记）

**当前行为**：每条 `NarrativeLine` 的 `links` 数组在渲染后永久存在，`<button>` 始终可点击。`executeCommand` 不检查该链接是否仍然有效。

**修复方案**：

**制作人已确认采用保留文字的方案**：
1. 在 `NarrativeLine` 接口中增加 `expired: boolean` 字段（默认 `false`）。
2. 在 `gameReducer` 中，当任何导致场景/状态变化的 action（如 `GOTO_LOCATION`、`EAT_FOOD`、`BUY_ITEM`）被触发时，将当前 `narrative` 数组中所有带 `links` 的历史记录标记为 `expired: true`。
3. 在 `NarrativeStream.tsx` 渲染时，对 `expired === true` 的记录：
   - **保留文字作为历史记录**，不可隐藏。
   - 将按钮状态设置为不可点击（`disabled` 或 `pointer-events-none`）。
   - **保留文字下划线**，仅改变颜色或透明度以示失效，但不移除下划线这一视觉特征。

## Bug-03：缺少饥饿值与体能值的联动机制

**问题描述**：当前体能值和饥饿值完全独立计算，体能没有自然恢复途径（除了喝矿泉水 +1），饥饿值也不影响体能。制作人要求饥饿值应能转化为体能值的限制或恢复条件。

**涉及文件**：
- `mvp-src/client/src/contexts/GameContext.tsx`
  - 第 296-308 行：`GT_TICK` 仅增加饥饿值，不处理体能恢复
  - 第 326-341 行：`COMPLETE_PROGRESS`（吃饭完成）仅降低饥饿值 5 点，不恢复体能
  - 第 210-227 行：`GOTO_LOCATION` 扣除体能（室内 1 点，室外 3 点），无恢复逻辑

**当前行为**：体能只会消耗（移动扣除），几乎没有恢复手段。饥饿值只会增加（`GT_TICK` 每分钟 +0.0694）和降低（吃饭 -5）。两者之间没有任何联动。

**修复方案（制作人已确认具体数值和阈值机制）**：

1. **进食恢复体能机制**：在 `COMPLETE_PROGRESS` 的 `eat_done` 分支中增加体能恢复逻辑。
   - **基础数值**：每次进食降低 `10` 点饥饿值，恢复 `2` 点体能值。
2. **饥饿阈值限制**：
   - **高饥饿状态（>80%）**：当饥饿值超过上限的 80% 时，进食只能降低饥饿值，**不能**恢复体能。
   - **低饥饿状态（<=50%）**：必须等到饥饿值消减到上限的 50% 以内时，进食才能同时降低饥饿值并恢复体能（-10 饥饿，+2 体能）。
   - **中等饥饿状态（50% < 饥饿 <= 80%）**：制作人未明确说明，默认按"不恢复体能"处理，即只有 `<=50%` 才能恢复体能。

*注：休息恢复体能的具体数值制作人暂未提及，当前仅实现进食联动机制。*

## Bug-04：缺少生命值（Health/HP）属性

**问题描述**：MVP 启动清单中明确要求实现"死亡/昏迷判定逻辑"，但当前代码中 `PlayerStats` 接口不包含生命值属性，状态栏也不显示生命值。

**涉及文件**：
- `mvp-src/client/src/contexts/GameContext.tsx`（第 4-18 行：`PlayerStats` 接口仅有 `stamina`、`hunger`、`money`、`skills`）
- `mvp-src/client/src/components/StatusBar.tsx`（第 36-44 行：仅显示体能、饥饿、金钱）

**当前行为**：不存在生命值概念，玩家无法"死亡"或"昏迷"。

**修复方案（制作人已确认具体数值和扣除规则）**：

1. **属性实装**：在 `PlayerStats` 接口中增加 `health` 和 `maxHealth`，**上限固定为 120**。
2. **UI 展示**：在 `StatusBar.tsx` 中增加生命值显示条。
3. **扣除触发条件与速率**：
   - **条件 1：满饥饿状态**。当饥饿值达到 100% 时开始扣除生命值。
   - **速率设定**：
     - "3 天不吃饭饥饿值就会满"（即饥饿值从 0 到上限需要 3 个游戏日）。
     - "满饥饿后再过 1 天半就会直接昏迷"（即满饥饿状态下，生命值从 120 扣到 0 需要 1.5 个游戏日）。
   - **条件 2：意外事件**（如被车撞等），当前阶段暂不实装，留作后续设计。
4. **归零逻辑**：生命值归零时，当前阶段统一处理为**"昏迷"**。
   - 昏迷后会被送进医院，需要交钱（具体逻辑参考相关文件）。
   - "死亡"逻辑当前不做，后续再设计。

## Bug-05：注册流程与游戏入口未分离（远期规划）

**问题描述**：当前每次刷新页面都从"边境站入境（Intro）"开始。制作人指出：未来接入后端后，注册入境流程应仅在首次创建角色时触发一次，后续登录应从主界面或上次下线位置继续。

**涉及文件**：
- `mvp-src/client/src/pages/Home.tsx`（第 21-29 行：`state.phase === 'intro'` 始终作为默认入口）
- `mvp-src/client/src/contexts/GameContext.tsx`（第 108 行：`initialState.phase` 硬编码为 `'intro'`）
- `mvp-src/client/src/main.tsx`（无登录/状态恢复逻辑）

**当前行为**：`GameProvider` 每次挂载时使用硬编码的 `initialState`，`phase` 始终从 `'intro'` 开始。没有任何持久化或状态恢复机制。

**修复方案（远期，需后端支持后实施）**：

1. **近期（前端模拟阶段）**：暂不修改。当前行为可接受，因为 MVP 阶段需要反复测试完整流程。
2. **远期（后端接入后）**：
   - 在 `Home.tsx` 中增加 `'login'` 和 `'main_menu'` 两个新的 `GamePhase`。
   - 默认入口改为 `'login'` → 登录成功后，从后端读取玩家数据 → 如果是新玩家则进入 `'intro'` → 如果是老玩家则直接进入 `'gameplay'`，坐标和状态从数据库恢复。
   - `initialState` 不再硬编码，而是由后端 API 返回。

**制作人已确认**：此项为远期规划，当前阶段不实施，待后端基础设施就绪后再推进。
