# 后端架构与 Tick 引擎规划（方向性方案）

本文档记录了个人开发者低成本后端架构的方向性规划。**本方案仅作记录和讨论，不在当前阶段立即实施。** 所有技术选型和部署决策需制作人最终确认。

技术栈沿用 `未细化暂定大纲.md` 中的架构共识（FastAPI + PostgreSQL + Redis + Docker）。

---

## 一、当前状态分析

### 现有后端

`mvp-src/server/index.ts` 当前仅为一个极简的 Express 静态文件服务器，用于在生产环境中托管前端构建产物。不包含任何游戏逻辑、数据库连接或 WebSocket 通信。

### 现有 Tick 引擎

游戏时间系统完全运行在前端 `GameContext.tsx` 中。`GT_TICK` 通过 `setInterval`（每 1000ms 触发一次）驱动游戏时间推进，每次 Tick 增加 1 GT 分钟并按固定速率增加饥饿值。所有属性计算、场景逻辑、物品交互均在前端 `useReducer` 中完成。

**核心问题**：关闭浏览器后游戏时间停止，无法实现离线消耗、全服同步和多人共存。

---

## 二、低成本部署策略（单机自建模式）

针对个人开发者的预算限制，放弃高昂的云原生 PaaS 和托管数据库服务，转向**国内轻量应用服务器的单机 Docker 部署方案**。

**服务器选择**：采用阿里云或腾讯云的"轻量应用服务器"（如 2核2G 或 2核4G 配置），针对个人开发者通常有较大折扣。

**部署方式**：在单台轻量服务器上使用 `Docker Compose` 一键编排 FastAPI 应用容器、PostgreSQL 数据库容器、Redis 缓存容器和 Nginx 反向代理容器。所有服务通过 Docker 内部网络通信，延迟极低，维护成本和心智负担较小，非常适合 MVP 到初期运营阶段。

## 三、Tick 引擎后端化方案

### 3.1 演进路径

**第一步（当前）**：前端 `setInterval` 模拟，仅用于 UI 验证和体验测试。

**第二步（后端 MVP）**：将 Tick 逻辑迁移到 FastAPI 后端。后端启动 `asyncio` 定时任务，按固定间隔执行 City Tick 结算，遍历所有在线玩家计算属性变化，通过 WebSocket 推送结算结果。前端不再自行计算属性，仅负责接收和渲染。

**第三步（离线补算）**：玩家下线时后端记录时间戳和状态快照到 PostgreSQL，上线时比对时间差一次性补算所有属性变化，通过"简报"界面展示离线期间事件摘要。服务器不需要 24 小时高频空转所有离线玩家的状态，大幅降低单机 CPU 负载。
