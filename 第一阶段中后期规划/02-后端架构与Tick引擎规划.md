# 后端架构与 Tick 引擎规划（方向性方案）

本文档记录了个人开发者低成本后端架构的方向性规划。**本方案仅作记录和讨论，不在当前阶段立即实施。** 所有技术选型和部署决策需制作人最终确认。

---

## 一、当前状态分析

### 现有后端

`mvp-src/server/index.ts` 当前仅为一个极简的 Express 静态文件服务器，用于在生产环境中托管前端构建产物。不包含任何游戏逻辑、数据库连接或 WebSocket 通信。

### 现有 Tick 引擎

游戏时间系统完全运行在前端 `GameContext.tsx` 中：
- `GT_TICK` 通过 `setInterval`（每 1000ms 触发一次）驱动游戏时间推进
- 每次 Tick 增加 1 GT 分钟，同时按固定速率增加饥饿值
- 所有属性计算、场景逻辑、物品交互均在前端 `useReducer` 中完成

**核心问题**：关闭浏览器后游戏时间停止，无法实现离线消耗、全服同步和多人共存。

## 二、目标后端架构

### 技术栈选型（沿用架构共识）

| 层级 | 技术选型 | 选择理由 |
| :--- | :--- | :--- |
| 后端框架 | Python (FastAPI) | 原生异步 + WebSocket 支持，适合 Tick 运算与实时推送 |
| 数据库 | PostgreSQL | 复杂 JSON 存储、强事务处理，适合双资产经济系统 |
| 缓存/队列 | Redis | 高速缓存 + Pub/Sub 机制，契合全服状态同步 |
| 前端通信 | WebSocket | 实时双向通信，替代当前的纯前端模拟 |
| 部署方案 | Docker Compose | 容器化编排，单机部署所有服务 |

### 低成本部署策略（单机自建模式）

针对个人开发者的预算限制，放弃高昂的云原生 PaaS 和托管数据库服务，转向**国内轻量应用服务器的单机 Docker 部署方案**。

**服务器选择**：采用阿里云或腾讯云的"轻量应用服务器"（如 2核2G 或 2核4G 配置）。这类服务器针对个人开发者通常有较大折扣，提供足够的带宽和基础性能。

**部署方式**：在单台轻量服务器上使用 `Docker Compose` 一键编排所有服务：
- FastAPI 应用容器
- PostgreSQL 数据库容器
- Redis 缓存容器
- Nginx 反向代理容器

所有服务通过 Docker 内部网络通信，延迟极低，维护成本和心智负担较小，非常适合 MVP 到初期运营阶段。

## 三、Tick 引擎后端化方案

### 3.1 从前端模拟到后端真实引擎的演进路径

**第一步（当前）**：前端 `setInterval` 模拟，仅用于 UI 验证和体验测试。

**第二步（后端 MVP）**：将 Tick 逻辑迁移到 FastAPI 后端：
- 后端启动一个 `asyncio` 定时任务，按固定间隔执行 City Tick 结算
- 每次 Tick 遍历所有在线玩家，计算属性变化（饥饿增加、体能消耗等）
- 通过 WebSocket 将结算结果推送给对应的前端客户端
- 前端不再自行计算属性，仅负责接收和渲染

**第三步（离线补算）**：实现离线数据补算机制：
- 玩家下线时，后端记录下线时间戳和当前状态快照到 PostgreSQL
- 玩家上线时，后端比对当前时间与下线时间戳，计算离线期间经过的 GT 时间
- 按照制作人定义的离线消耗规则（待提供），一次性补算所有属性变化
- 通过"简报"界面向玩家展示离线期间发生的事件摘要
- **优势**：服务器不需要 24 小时高频空转所有离线玩家的状态，大幅降低单机 CPU 负载

### 3.3 游戏时间的核心问题说明

制作人已明确指出：游戏时间是一个大问题，但因为没有后端所以不能始终在后台计算。这个问题的最终解决依赖于后端基础设施的建立。在当前前端模拟阶段，游戏时间仅在浏览器打开时推进，这是已知的技术限制，不影响 UI 和机制的验证工作。

## 四、后端重构的前置条件

在正式启动后端开发之前，以下条件需要满足：

| 前置条件 | 状态 | 说明 |
| :--- | :--- | :--- |
| 前端 UI 和核心机制通过制作人验证 | 进行中 | Bug-01 到 Bug-04 修复完成后 |
| 制作人提供 MVP 启动数值数据 | 待提供 | 体能上限、饥饿上限、消耗速率等 |
| 制作人确认离线消耗规则 | 待提供 | 离线期间属性如何变化 |
| 制作人确认部署方案和预算 | 待确认 | 选择具体的云服务商和配置 |

## 五、需要制作人确认的事项

1. 上述技术栈选型是否认可？（FastAPI + PostgreSQL + Redis + Docker）
2. 部署方案是否倾向国内轻量服务器单机自建？
3. 离线补算机制的方向是否认可？（上线时一次性补算，而非服务器持续空转）
4. 后端开发的启动时机：是在前端 Bug 全部修复后，还是可以并行推进？
