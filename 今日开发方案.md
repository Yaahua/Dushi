## 二、今日开发流程规划

基于 `dev-roadmap.md` 的第一阶段目标与中后期规划，今日的开发流程建议分为以下几个步骤推进：

目前代码中的场景为"世界公寓"（包含大厅、厨房、便利店等），而 MVP 启动清单中要求的是"火车站隔离区"（包含入境、服务中心、便利店、收容所）。

*   **行动项**：确认是否在今日将场景重构为"火车站隔离区"。
*   **前置需求**：若确认重构，需要制作人提供火车站隔离区各 POI 的描述文案、移动时间消耗（GT 分钟）与体能消耗数值。

### 步骤 2：后端基础设施搭建（第一阶段 1.1 - 1.3）

目前 `server/index.ts` 仅为静态文件服务器，尚未实现真正的后端逻辑。

*   **行动项**：
    1.  搭建 FastAPI 后端框架结构。
    2.  配置 PostgreSQL 数据库与 Redis 缓存连接（可先使用 Docker Compose 在本地环境搭建）。
    3.  建立前后端 WebSocket 通信机制，实现基础的连接与消息收发。
*   **审查节点**：向制作人展示后端项目结构与数据库/WebSocket 连接测试结果。

### 步骤 3：Tick 引擎后端化迁移（第一阶段 1.4 - 1.5）

将目前在前端 `GameContext` 中通过 `setInterval` 模拟的 Tick 引擎迁移至后端。

*   **行动项**：

    2.  实现 City Tick 定时结算逻辑（属性扣除、状态更新）。
    3.  通过 WebSocket 将结算结果推送到前端，前端改为纯渲染层。
*   **审查节点**：向制作人演示后端驱动的结算流程与前端状态同步效果。

## 三、关于"造轮子"与脚手架的建议

针对您提出的"需不需要造轮子脚手架"的问题，我的建议如下：

### 1. 前端脚手架

**不需要重新造轮子**。目前项目已经使用了 Vite + React + TypeScript + TailwindCSS 的现代化技术栈，且 UI 组件库（如 `components/ui/` 下的各类组件）已经相对完善。我们应继续沿用现有的前端架构，专注于业务逻辑与组件的优化。

### 2. 后端脚手架

**建议使用标准框架，不建议从零手写底层**。
*   **Web 框架**：直接使用 **FastAPI**。它原生支持异步和 WebSocket，非常适合我们 Tick 引擎和实时推送的需求。
*   **ORM**：建议使用 **SQLAlchemy** 或 **SQLModel** 配合 PostgreSQL，避免手写大量 SQL 语句，提高开发效率与安全性。
*   **缓存与队列**：使用 **redis-py** 连接 Redis。

### 3. 部署与环境脚手架

**强烈建议使用 Docker Compose**。
*   为了保证开发环境与未来生产环境的一致性，建议编写一个 `docker-compose.yml` 文件，一键启动 PostgreSQL 和 Redis 服务。这不仅方便本地开发测试，也符合《02-后端架构与Tick引擎规划.md》中提到的低成本单机部署策略。
